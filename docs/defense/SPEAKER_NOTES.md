## Slide 1 — An AI-Driven Open-Source Tool for the Design and Optimization of Solar Cells

TALK TRACK (60s): State the thesis in one sentence: an open-source, multi-technology solar-cell design platform in which every number is verifiable — by hand mathematics, by independent code paths, by certified fabricated devices, and by the incumbent tools. The four cards are the whole defense in miniature. Offer immediately: any of these can be regenerated live in under a minute.

DEEP BACKGROUND: Stack = Python 3.10+, NumPy/SciPy numerics, Streamlit UI, PyTorch for PINNs, scikit-learn for surrogates; ~15k lines, 58 regression tests. "rev B" denotes the cross-tool validation layer added on top of the v4.0 multi-technology release.

LIKELY QUESTIONS (CS):
Q: Why Python for numerical simulation — isn't it slow?  A: The hot loops are vectorized NumPy (compiled C under the hood); the fast simulator answers in milliseconds and the FDM solver in seconds. Development velocity, ecosystem (SciPy optimizers, autodiff via PyTorch) and auditability outweigh raw C++ speed at this problem size (1-D meshes of 10^2-10^3 nodes). Profiling showed >90% of time inside BLAS/LAPACK calls anyway.
Q: Why Streamlit rather than a web framework?  A: The UI is a thin, stateless view over pure-Python engines; every UI action maps to an importable function, so the science is testable without the UI. Streamlit gives reactive widgets for free and deploys from the GitHub repo directly.
Q: What does "verifiable" mean operationally?  A: Three scripts write versioned JSON artifacts; the UI renders those artifacts; tests assert against the same JSON. Single source of truth — a claim cannot drift from its evidence.

## Slide 3 — Why Another Solar-Cell Simulator?

TALK TRACK (90s): Three pain points. Fragmentation: no open tool designs a perovskite/Si tandem end-to-end. Provenance: parameters in published simulations rarely cite sources, so results cannot be audited — this is a data-provenance problem CS people recognize from reproducibility crises elsewhere. Validation culture: tools are graded against other tools, which can share systematic errors (correlated failure modes); certified fabricated devices are the only independent oracle.

DEEP BACKGROUND: The right-hand cards are design invariants, not aspirations: the '0 hand-typed numbers' claim is enforced structurally — UI pages render artifact JSON, and tests fail if a benchmark cites an unregistered reference.

LIKELY QUESTIONS (CS):
Q: Isn't 'validated against experiments' just integration testing with physical data?  A: Exactly — and that is the contribution: treating certified devices as a held-out test set with pre-registered tolerances, instead of the field's habit of qualitative agreement plots.
Q: How do you prevent overfitting the tool to those 14 devices?  A: Separation of calibration and test roles is explicit per benchmark (e.g., organic 300 nm point is out-of-calibration), tolerances are fixed before runs, and the community-contribution path continuously grows the test set with devices we never tuned on.
Q: What's the threat model for provenance — can a contributor inject a fake DOI?  A: The registry-integrity test enforces schema (DOI prefix, citation length, confidence tier), and audited builds cross-check entries against the Consensus academic index; PRs adding benchmarks are reviewed against the cited paper.

## Slide 6 — The Four-Layer Verification Ladder

TALK TRACK (90s): This is the methodological heart. Rungs are ordered by independence from my own code: hand mathematics (no code), redundant implementations (my code vs my code, different algorithms), certified devices (my code vs nature), external tools (my code vs their code). The key inversion: fabricated devices outrank tool-to-tool comparison because two simulators can share correlated systematic errors — think of it as N-version programming: agreement between versions is only evidence if failures are independent, which for physics engines trained on the same literature they are not.

LIKELY QUESTIONS (CS):
Q: Rung 2 is just redundancy — what failure classes does it actually catch?  A: Implementation bugs (sign errors, unit slips, indexing), discretization artifacts, and library regressions — anything that breaks one path but not an algorithmically different one. It cannot catch shared modeling error; that is rung 3's job.
Q: How is rung 1 automated if it's 'by hand'?  A: The hand derivations are frozen into run_tutorial_examples.py as assertions with explicit closed-form arithmetic — the script is the notarized version of the whiteboard.
Q: What is your regression strategy when physics code changes?  A: 58 pytest tests pin benchmark errors, round trips, thermodynamic bounds and API contracts; any commit that moves a validated number outside tolerance fails CI.

## Slide 8 — Engine I — Perovskite: Full Drift-Diffusion + Calibrated Fast Solver

TALK TRACK (90s): Perovskite engine has two solvers by design. FDM: Poisson + electron/hole continuity, Scharfetter-Gummel (SG) exponential-fitting fluxes, Newton on the coupled system — the SCAPS-class method. Fast solver: analytic transport/recombination calibrated on the published-SCAPS suite; milliseconds, used for sweeps and optimization. Fig: Nt is the quality knob — PCE collapses over three decades of defect density, which is the SRH physics a CS person can read as a log-linear sensitivity.

DEEP BACKGROUND for CS questions: SG answers a stiffness problem: carrier densities vary over ~15 orders of magnitude across nm distances; naive central differences on drift-diffusion go unstable unless mesh Peclet < 2, which would need absurd meshes. SG fits the exact solution of the local constant-field two-point BVP (Bernoulli functions B(x)=x/(e^x−1)), giving unconditional stability. Solved in scaled variables (thermal voltage, intrinsic density, Debye length) to condition the Jacobian; Gummel iteration (block Gauss-Seidel over psi, n, p) provides robust warm starts, Newton finishes quadratically.

LIKELY QUESTIONS (CS):
Q: Complexity?  A: Tridiagonal blocks -> O(N) per Newton step via Thomas algorithm; N ~ few hundred nodes; a full J-V sweep is O(N·V·iters), well under a second.
Q: Convergence guarantees?  A: None global (the system is nonlinear); practically, damped Newton + voltage-continuation from equilibrium converges everywhere in the validated parameter box; divergence outside it is reported, not silently absorbed.
Q: Why keep two solvers rather than one?  A: Speed/fidelity Pareto: the fast solver's error vs FDM is itself a tested quantity — a classic surrogate-with-certified-error pattern.

## Slide 9 — Engine II — Crystalline Silicon: Implied-Voc with Certified Presets

TALK TRACK (90s): Silicon engine is analytical, not FDM — deliberately: for wafer-scale c-Si the community's own gold-standard analyses (Kerr-Cuevas implied-Voc, Richter parameterization) are algebraic balances, and the certified records constrain them tightly. Emphasize the hand-verifiable chain: Δn from a 1-D root-find (Brent), Voc from the np product, FF from Green's formula, then a full Newton diode sweep. Table: five architectures, all ≤1.9% from certified devices.

LIKELY QUESTIONS (CS):
Q: Brent vs Newton for the balance — why?  A: The generation-recombination balance is monotone in Δn but its derivative spans decades; Brent is derivative-free, bracketed, and guaranteed to converge — robustness over speed for a scalar root.
Q: The Richter/Niewelt switch looks like a strategy pattern — is model choice a config?  A: Yes: intrinsic-recombination models are injectable functions; the 0.56% PCE deviation between them is itself a regression test (model-form uncertainty made explicit).
Q: The 29.4% guard — how do you test a 'no configuration exceeds X' property?  A: A property-style test drives a deliberately utopian architecture (infinite lifetime, zero resistance, perfect optics) and asserts the ceiling; it's the physics analogue of a fuzzed invariant.
Q: Where do preset parameters come from — fitted or cited?  A: Each preset's J0s/tau/Rs sit inside published ranges for that architecture and are then fixed; the benchmark table is evaluation, not re-fitting.

## Slide 11 — Engine IV — Tandem: True Series J-V Coupling

TALK TRACK (90s): Tandem coupling is an algorithmic contribution: invert each subcell J-V to V(J) on the photocurrent branch (monotone segment; deduplicated interpolation), sum voltages on a common current grid, subtract interconnect I·R, take the max-power point of the summed curve. This is physically superior to min-Jsc heuristics because the non-limiting subcell operates off its own MPP — the summed curve captures that automatically. Spectral filtering: Beer-Lambert of the top absorber times interconnect/parasitic factors, with a texture path factor applied consistently to top absorption and bottom transmission (photon bookkeeping conserved).

LIKELY QUESTIONS (CS):
Q: Numerical pitfalls of inverting J-V?  A: The curve must be strictly monotone on the branch used; we sort, deduplicate, and restrict to J≥small-negative before interp — a unit test covers a pathological flat-FF curve.
Q: Complexity of the matching scan?  A: O(K) tandem solves for K thicknesses; each solve is two engine calls + O(G) interpolation on a 400-point grid — sub-second per point, so exhaustive scan beats anything fancier at this dimensionality.
Q: Why does mismatched FF exceed matched FF — bug or physics?  A: Physics: the bottom cell below its Jsc contributes near-flat voltage; documented, and bounded by the interconnect Rs term calibrated per record generation.

## Slide 15 — Validation III — Lumerical-Class Wave Optics (TMM)

TALK TRACK (90s): The Lumerical answer, precisely: for planar multilayers, Lumerical STACK (and FDTD in the planar limit) solves the coherent Maxwell problem whose exact 1-D solution is the transfer-matrix method. So the validation target is the published TMM benchmark literature — and we hit it: interference maxima inside every published window, energy conservation <1e-4, Fresnel to machine precision, absolute certified Jsc within 5.1% where the published TMM-vs-experiment band averages ~16% (Rosa 2021).

DEEP BACKGROUND (algorithm): per wavelength, 2x2 complex interface/phase matrices; internal fields from partial-system matrices; per-layer absorption by integrating the dissipation density (4πnk/λn0)|E(x)|². Complexity O(N_λ · N_layers · N_x); the whole spectral solve is milliseconds. During development the energy-conservation check caught a real bug — a spurious e^{iξd} phase factor inflating fields in absorbing layers — which is rung-1 verification doing its job.

LIKELY QUESTIONS (CS):
Q: Why not just run Lumerical and diff?  A: License-free reproducibility: the published TMM benchmarks ARE the community's Lumerical-class results, and matching them is the same statement without a commercial dependency in the pipeline. The export path exists for anyone who owns a license.
Q: Floating-point: 'machine precision' claims?  A: Fresnel matches analytic to ~1e-16 (double); conservation residual 1e-4 is quadrature error of the field integral, shown to shrink with x-resolution — the distinction is stated.
Q: n,k data provenance?  A: Tabulated from cited ellipsometry literature with disclosed ±5-10% batch spread; tables live in code with DOIs, swappable for user measurements.

## Slide 17 — AI I — Conditional PINN with an Explicit Verification Protocol

TALK TRACK (2 min — expect the most CS questions here): Conditional PINN: one MLP u_theta(x; d, Nt) over a device family, trained on FDM solutions plus PDE residuals via automatic differentiation. Value = millisecond field inference inside optimization/inverse-design loops. Verification protocol has three legs: (a) held-out devices vs FDM — 6.5-7.6% relative-L2; (b) closed-form anchors (full-depletion uniform field; dark-equilibrium np=ni²) that need no reference solution; (c) every candidate re-simulated through FDM before being reported.

DEEP BACKGROUND: Inputs scaled to [0,1]; outputs are (psi, log n, log p) — logs because densities span ~15 decades and L2 on raw densities would see only the peak. Loss = MSE_data + λ·(Poisson residual) + continuity terms; λ balanced so gradient norms are commensurate (the standard PINN loss-balancing issue). Training on GPU minutes/CPU tens of minutes; weights + metrics ship as artifacts with fixed seeds.

LIKELY QUESTIONS (CS):
Q: Why PINN instead of a plain neural surrogate on (d,Nt)→J-V?  A: We ship those too (ml_models). The PINN's residual term regularizes toward physics between training points and yields full field profiles, enabling analytic-anchor checks a black-box cannot pass or fail meaningfully.
Q: Spectral bias / boundary layers — can an MLP represent SG-sharp junction fields?  A: Partially — that is exactly why held-out error is 6-8% not 0.1%, why errors concentrate at junctions (shown in thesis), and why the FDM remains ground truth. The PINN is a fast index into solution space, never the oracle.
Q: Generalization outside the training box?  A: Explicitly unsupported: the box (300-700 nm, 1e13-1e16) is stated; extrapolation metrics are reported separately and worse; the UI labels out-of-family queries. Protocol (c) makes extrapolation errors non-fatal — FDM re-simulation gates every reported result.
Q: Why rel-L2 as the metric?  A: Scale-free across devices and fields; complemented by the physics anchors, which are max-norm-style pointwise checks. Q: Reproducible training?  A: Seeded, pinned versions, metrics JSON in-repo; retraining reproduces held-out errors within run-to-run noise (<0.5% absolute).

## Slide 18 — AI II — Evolutionary Optimization with Sensitivity Explanation

TALK TRACK (90s): Optimization slide shows a live run, not a schematic: DE lifts industrial PERC 23.60% → 25.70% in 604 evaluations; the random-forest importances attribute the gain to surface passivation (J0s) and bulk lifetime — the 'why' next to the 'what'.

DEEP BACKGROUND: scipy differential_evolution, rand/1/bin-style mutation x' = xa + F(xb−xc), population 10-12 per dimension-ish, deferred updating for parallel evaluation, seeded. Sensitivity: RF fitted on the evaluated population (X→PCE), impurity importances reported.

LIKELY QUESTIONS (CS):
Q: DE has no convergence guarantee — how do you claim an optimum?  A: We do not claim global optimality; we claim best-found under budget with the convergence trace shown, multi-seed agreement in the benchmark page, and physical bounds enforced. For 3-6 dimensional boxes, DE's empirical reliability is well documented — and the benchmark page races it against GA/NSGA-II/Bayesian on known optima.
Q: No-free-lunch — why DE as default?  A: Measured, not assumed: page 6 gives equal-budget comparisons on this landscape class; DE wins or ties on robustness; Bayesian wins on tiny budgets — stated there.
Q: RF impurity importance is biased toward high-cardinality/correlated features.  A: Correct — here all features are continuous scalars on comparable ranges and the ranking is triangulated against one-at-a-time physical sweeps; permutation importance gives the same ordering (thesis appendix). It is an explanation aid, not an inferential claim.
Q: The optimizer found 25.7% 'PERC' — is that physical?  A: It found what the parameters permit: J0s=2 fA is TOPCon-grade passivation, i.e., the optimizer rediscovered the PERC→TOPCon transition. Boundary optima are read as roadmap statements, and hard thermodynamic guards (29.4%) are enforced in code.
